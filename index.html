Agent ① 需求解析

模糊的自然语言需求 → 结构化看板规格书

📥 输入
自然语言需求 数据源信息 口径说明 表间关联关系
📤 输出
看板规格书 (JSON) 指标清单 维度需求 筛选需求 隐含需求标注
内部处理流程
Step 1: 意图识别
解析分析师的自然语言描述，提取核心意图
LLM 推理
"搭一个电商大促日报看板"
→ 识别主题：电商大促，类型：日报，目标：看板
Step 2: 指标/维度提取
从需求和口径说明中提取所有需要的指标和维度
规则 + LLM 混合
需求中提及的指标 → 直接提取
口径说明中的衍生指标 → 关联提取
隐含指标 → 推理补充（如"日报"→ 同比/环比）
Step 3: 隐含需求推理
根据业务场景自动补充分析师没说但需要的分析维度
行业知识库 + LLM
"日报" → 自动添加 day_over_day、week_over_week
"大促" → 自动添加 活动期 vs 日常期对比
"GMV趋势" → 自动添加 移动平均线
Step 4: 口径完整性检查
对照口径说明，检查是否所有需要的指标都有明确定义
◆ 需求中提到的指标在口径说明中是否有定义？
✓ 完整 → 继续 ✗ 缺失 → 标记待确认
⚠️ 例：渠道ROI 需求中提到但口径说明中未定义
→ 在规格书中标记为 "待确认口径"，不阻塞后续流程
Step 5: 输出看板规格书
生成结构化 JSON，包含看板名称、分区块、每个区块的指标/维度/图表类型
{"dashboard_name", "refresh_freq", "global_filters", "sections": [{"title", "metrics", "dimensions", "chart_type", "compare"}]}
🏗️

Agent ② 语义模型

数据源 + 口径 → 语义模型定义 + SQL + 自定义指标

📥 输入
数据源（表名+字段） 表间关联关系 口径说明 看板规格书（来自①）
📤 输出
语义模型SQL 维度识别+调整建议 度量识别+调整建议 自定义指标 (CASE WHEN) 模型拆分建议
内部处理流程
Step 1: JOIN关系构建
根据用户提供的关联关系，生成多表 JOIN SQL
规则引擎 + LLM
用户提供关联 → 直接按 JOIN 条件生成
未提供 → LLM 根据字段名推理（如 channel_id）→ 请用户确认

JOIN 类型决策：
· 维度表（dim_xxx）→ LEFT JOIN（保留主表数据）
· 事实明细表 → INNER JOIN
· 退款/评价等稀疏表 → LEFT JOIN
Step 2: WHERE 过滤条件生成
根据口径说明，在 SQL 中添加模型级的过滤条件
规则引擎
· 口径中的时间范围 → WHERE dt BETWEEN ...
· 排除无效数据 → WHERE order_status != 'cancelled'
· 多个条件 → AND 组合

⚠️ 注意：模型级过滤影响所有度量，仅放置通用过滤；
度量级差异过滤放 Step 4 的 CASE WHEN 中
Step 3: 模型复杂度评估 & 拆分建议
评估单条 SQL 是否过于复杂，必要时建议拆分为多个语义模型
◆ 关联表数量 > 5 或口径差异大？
建议拆分 单模型
拆分场景示例：
· 口径过滤条件差异大（如 GMV 仅 paid，订单量排除 cancelled）
→ 方案：统一用最宽过滤，各指标内 CASE WHEN 补过滤

· 关联表太多（>5 张）
→ 方案：拆成 2-3 个语义模型（如订单模型 + 退款模型）
Step 4: 维度/指标自动识别 + 调整建议
模拟 BI 平台自动解析，并标注需要人工调整的项
规则引擎 + LLM

维度分类规则：
· STRING + 低基数 → 维度候选
· 含 date/dt/time → 日期维度，自动推导粒度（日/周/月）
· 含 ID 后缀 → ⚠️ 系统可能误判为维度，需手动移除

度量分类规则：
· NUMERIC → 度量候选（SUM/AVG）
· 含 ID 后缀 → COUNT(DISTINCT)
· 特殊字段 → 需人工确认（如 order_status 可能是维度也可能是过滤条件）
Step 5: 自定义/复杂指标生成
基于口径说明生成需要手工添加的自定义指标
LLM + 规则后处理

处理策略：
· CASE WHEN 过滤 → SUM(CASE WHEN status='paid' THEN amount ELSE 0 END)
· 指标间引用 → [净GMV] / NULLIF([有效订单量], 0)，标注依赖
· 窗口函数 → 当日 / LAG(当日, 7)，生成 CTE
· 任何除法 → 统一加 NULLIF(..., 0) 防止除零

依赖排序：先定义基础度量，再定义引用型衍生指标
Step 6: 输出语义模型搭建指南
汇总以上步骤，输出完整的语义模型搭建文档
文档包含：
· 语义模型 SQL（多表 JOIN + WHERE）
· 维度识别结果 + 调整建议
· 度量识别结果 + 调整建议
· 自定义指标（含 CASE WHEN 表达式）
· 性能/拆分注意事项
📊

Agent ③ 图表设计

看板规格书 + 语义模型定义 → 图表选型 + SQL + 布局 + 交互

📥 输入
看板规格书（来自①） 语义模型定义（来自②） 可用维度/度量列表
📤 输出
看板布局方案 每个图表配置详情 图表级SQL 筛选器配置
内部处理流程
Step 1: 规格书 ↔ 语义模型 对齐检查
检查需求中的每个指标是否都在语义模型中有对应度量定义
◆ 需求指标在语义模型中是否都有对应度量？
✓ 全部匹配 → 继续 ✗ 缺失 → 回退请求补充
例：需求要"退款率"，但语义模型中只有 refund_amount 和 pay_amount
→ 退款率是衍生指标，需在语义模型中添加（回退到 Agent ② 补充）
Step 2: 图表选型
为看板规格书中的每个区块推荐最合适的图表类型
确定性规则（优先命中）
· 趋势/日变化 → 折线图
· 对比/TOP N → 柱状图
· 占比/分布 → 饼图/环形图
· 单个KPI → 指标卡
· 层级占比 → 矩形树图

LLM 兜底（规则未命中时）
分析指标+维度的数据特征 → 推荐最合适图表
Step 3: 图表 SQL 生成
为每个图表生成独立的查询 SQL（基于语义模型SQL + 图表维度/度量）
基于语义模型SQL扩展
· 复用语义模型的 FROM ... JOIN ... WHERE
· 替换 SELECT 为当前图表所需的维度+度量
· 追加图表级筛选条件（如品类 = '电子产品'）
· 添加排序（ORDER BY）和限制（TOP N）

多语义模型场景：明确标注每个图表使用哪个语义模型
Step 4: 布局编排
规划看板的整体布局结构
布局规则引擎
· 第一行：核心KPI指标卡（一行 3-4 个）
· 第二行：趋势图 + 对比图（2列布局）
· 第三行：分布图/明细表
· 高优先级指标 → 占据更大面积
· 同维度图表 → 放在同一行便于联动
Step 5: 筛选器 & 联动设计
配置全局筛选器、图表间联动关系
筛选器规则
· 日期维度 → 默认加日期范围筛选器
· 分类字段（唯一值 < 50）→ 可选筛选器
· 同维度图表 → 点击联动过滤
· 筛选器类型：单选 / 多选 / 日期范围 / 搜索下拉
Step 6: 样式建议
为每个图表提供基础样式配置建议
· 指标卡：数字格式、同比颜色（涨红跌绿）
· 折线图：是否需要双Y轴、数据标签、平滑曲线
· 柱状图：排序方式、颜色方案、是否堆叠
· 饼图：TOP N + 其他、标签显示
📋

Agent ④ 方案生成

汇总所有 Agent 的产出 → 输出完整、可直接执行的搭建方案文档

📥 输入
看板规格书（来自①） 语义模型指南（来自②） 图表设计方案（来自③）
📤 输出
📄 语义模型搭建指南.md 📄 看板搭建指南.md
内部处理流程
Step 1: 信息汇总 & 去重
合并三个 Agent 的输出，去除重复内容，统一术语
规则处理
· 语义模型中的度量名称 ↔ 看板图表中引用的指标名称 → 统一
· 维度在多处出现 → 以语义模型定义为准
· 口径说明 → 只在语义模型指南中出现一次，看板指南中引用
Step 2: 生成文档 A — 语义模型搭建指南
组织语义模型相关内容为独立文档
文档结构：
1. 模型基本信息（名称、数据源、基础过滤）
2. 语义模型 SQL（多表 JOIN + WHERE）
3. 维度识别表 + 调整建议
4. 度量识别表 + 调整建议
5. 自定义指标（含完整 CASE WHEN 表达式）
6. 注意事项（性能、拆分建议）
Step 3: 生成文档 B — 看板搭建指南
组织看板相关内容为独立文档
文档结构：
1. 看板概览（名称、刷新频率、使用的语义模型）
2. 布局图（ASCII 或文字描述）
3. 全局筛选器配置
4. 各图表详细配置：
  · 图表类型 & 使用的语义模型
  · 维度 & 度量
  · 图表级 SQL
  · 样式 & 交互配置
  · 联动规则
5. 注意事项
Step 4: 质量自检
在输出前进行一致性校验
校验规则
· 看板指南中引用的每个度量是否在语义模型指南中有定义
· 看板指南中引用的每个维度是否在语义模型 SQL 的 SELECT 中存在
· SQL 语法是否正确（字段名、表名与输入一致）
· 图表选型是否与数据特征匹配
· 是否有遗漏的指标或筛选需求
Step 5: 确认项汇总 → 人工交接
合并所有 Agent 标记的待确认项，在方案文档末尾输出，交由人工审核
◆ 确认项是否全部通过？
✓ 通过 → 执行搭建 ✗ 需修正 → 反馈调整
人机协作交接点

确认项来源：各 Agent 在推理过程中标记的无法自主决策的点

确认项分类：
· 🔴 口径确认 — 业务定义类（如"GMV是否含退款"）
· 🟡 数据可行性 — 表结构/字段是否存在（一期无元数据服务，需人工验证）
· 🔵 过滤条件 — 业务规则类（如"是否排除测试订单"）

处理方式：
· 一期：确认项写入方案文档末尾，人工逐项审核，反馈后重新生成
· 二期：接入元数据服务后，数据可行性类可自动校验，确认项大幅减少
· 三期：BI平台API打通，方案直接落库，人工仅需最终审核
