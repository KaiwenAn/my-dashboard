二 语义模型搭建指南

🏗️

模型：dm_optimize_event_cost_daily

日粒度可优化事件成本分析模型，聚合所有维度与指标，支持按App、事件、地域、部门等多维度下钻分析

{} SQL 
-- 手机部可优化事件成本日报语义模型
-- 覆盖指标：可优化成本、charge_track、charge_use
-- 数据粒度：事件行（日维度通过主表date字段聚合）
-- 核心逻辑：仅对与dwm_bill_optimize_event_v2_df关联成功的事件行计算可优化成本

WITH
-- Step 1: 维度表预处理：dwm_bill_event_onetrack_charge_share_di 作为维度表使用时，按 app_id, date 去重
-- 避免因粒度不一致导致主表数据膨胀
onetrack_dept_dedup AS (
    SELECT DISTINCT
        app_id,
        date,
        bu_dept1_name,
        bu_dept2_name
    FROM iceberg_zjyprc_hadoop.meta.dwm_bill_event_onetrack_charge_share_di
    WHERE 1=1
        -- 模型级过滤：日期同步到该表
        AND date = '${date-1}'
),

-- Step 2: 可优化事件表预处理：dwm_bill_optimize_event_v2_df 也需同步日期过滤
optimize_events AS (
    SELECT
        event_guid,
        date,
        opt_flg
    FROM iceberg_zjyprc_hadoop.meta.dwm_bill_optimize_event_v2_df
    WHERE 1=1
        -- 模型级过滤：日期同步到该表
        AND date = '${date-1}'
)

SELECT
    -- ===== 维度字段 =====
    -- 日期维度（来自主表）
    e.date AS date,

    -- app_id（来自主表）
    e.app_id AS app_id,

    -- app_name（来自track_app_info.name字段）
    -- ⚠️ TODO: 需确认 - 上游key_fields中未找到'app_name'字段，此处使用name字段
    app_info.name AS app_name,

    -- app_owner（来自track_app_info.owner字段）
    -- ⚠️ TODO: 需确认 - 上游key_fields中未找到'app_owner'字段，此处使用owner字段
    app_info.owner AS app_owner,

    -- region（来自主表）
    e.region AS region,

    -- event_name（来自主表event_name字段）
    e.event_name AS event_name,

    -- event_name_cn（来自track_event_info.display_name字段）
    -- ⚠️ TODO: 需确认 - 上游key_fields中未找到'event_name_cn'字段，此处使用display_name字段
    event_info.display_name AS event_name_cn,

    -- event_guid（来自主表）
    e.event_guid AS event_guid,

    -- opt_flg（来自dwm_bill_optimize_event_v2_df）
    opt.opt_flg AS opt_flg,

    -- bu_dept1_name（来自去重后的onetrack部门维度表）
    dept.bu_dept1_name AS bu_dept1_name,

    -- bu_dept2_name（来自去重后的onetrack部门维度表）
    dept.bu_dept2_name AS bu_dept2_name,

    -- ===== 原子指标 =====
    -- charge_track: 事件采集成本
    SUM(e.charge_track) AS charge_track,

    -- charge_use: 事件使用成本
    SUM(e.charge_use) AS charge_use,

    -- ===== 衍生指标 =====
    -- 可优化成本: 仅对与dwm_bill_optimize_event_v2_df关联成功的事件行求和
    -- 关联成功条件：event_guid匹配且date匹配（通过LEFT JOIN + IS NOT NULL判断）
    -- 注意：dwm_bill_optimize_event_v2_df表中所有事件均为可优化事件，无需用opt_flg过滤
    SUM(CASE WHEN opt.event_guid IS NOT NULL THEN e.charge_track + e.charge_use ELSE 0 END) AS optimize_cost

FROM iceberg_zjyprc_hadoop.meta.dm_bill_event_charge_without_visit_v2_df e

-- LEFT JOIN track_app_info: 获取app名称和负责人
-- 关联条件：e.app_id = app_info.real_app_id（使用field_mappings中的实际字段名real_app_id）
LEFT JOIN iceberg_zjyprc_hadoop.tracking.track_app_info app_info
    ON e.app_id = app_info.real_app_id

-- LEFT JOIN track_event_info: 获取事件中文名
-- 关联条件：e.event_name = event_info.name（使用field_mappings中的实际字段名name）
-- 注意：track_event_info的is_delete过滤条件仅应用于该表，不影响主表行数
LEFT JOIN iceberg_zjyprc_hadoop.tracking.track_event_info event_info
    ON e.event_name = event_info.name
    AND event_info.is_delete = '0'

-- LEFT JOIN dwm_bill_optimize_event_v2_df: 标识可优化事件
-- 关联条件：event_guid和date双字段匹配
-- 此表所有事件均为可优化事件，无需用opt_flg过滤
LEFT JOIN optimize_events opt
    ON e.event_guid = opt.event_guid
    AND e.date = opt.date

-- LEFT JOIN 去重后的onetrack部门维度表: 获取部门归属
-- 关联条件：app_id和date双字段匹配
-- 此表作为维度表使用，已按app_id,date去重
LEFT JOIN onetrack_dept_dedup dept
    ON e.app_id = dept.app_id
    AND e.date = dept.date

WHERE 1=1
    -- 模型级过滤：日期过滤，同步应用到所有包含date字段的表
    -- 主表日期过滤
    AND e.date = '${date-1}'
    -- 模型级过滤：部门过滤，仅展示手机部数据
    AND dept.bu_dept1_name = '手机部'
    -- 模型级过滤：排除app_id为NULL的数据
    AND e.app_id IS NOT NULL

GROUP BY
    e.date,
    e.app_id,
    app_info.name,
    app_info.owner,
    e.region,
    e.event_name,
    event_info.display_name,
    e.event_guid,
    opt.opt_flg,
    dept.bu_dept1_name,
    dept.bu_dept2_name

CTE预处理： > - onetrack_dept_dedup：将dwm_bill_event_onetrack_charge_share_di按app_id,date去重，避免多对多JOIN导致主表数据膨胀。同时在此CTE中应用日期过滤（date = '${date-1}'） > - optimize_events：将dwm_bill_optimize_event_v2_df预过滤日期，减少JOIN数据量 > > 2. 主表与维度表关联： > - 主表 dm_bill_event_charge_without_visit_v2_df 作为事实表，LEFT JOIN所有维度表 > - track_app_info：通过 e.app_id = app_info.real_app_id 关联（使用field_mappings中的实际字段名real_app_id） > - track_event_info：通过 e.event_name = event_info.name 关联（使用field_mappings中的实际字段名name），并应用is_delete='0'过滤 > - dwm_bill_optimize_event_v2_df：通过 e.event_guid = opt.event_guid AND e.date = opt.date 双字段关联 > - dwm_bill_event_onetrack_charge_share_di：通过 e.app_id = dept.app_id AND e.date = dept.date 双字段关联（使用去重后的CTE） > > 3. 指标计算： > - charge_track/charge_use：直接SUM聚合 > - 可优化成本：使用CASE WHEN判断opt.event_guid IS NOT NULL（即与可优化事件表关联成功），对关联成功的事件行求和charge_track+charge_use > > 4. WHERE过滤： > - 日期过滤：同步应用到主表（e.date）和onetrack部门维度表（已在CTE中应用） > - 部门过滤：dept.bu_dept1_name = '手机部' > - app_id非空过滤：e.app_id IS NOT NULL > - track_event_info的is_delete过滤：在LEFT JOIN的ON条件中应用，不影响主表行数

关联逻辑

左表关联方式右表关联条件原因
`dm_bill_event_charge_without_visit_v2_df`LEFT JOIN`track_app_info``e.app_id = app_info.real_app_id`获取app名称和负责人信息。使用field_mappings中的实际字段名real_app_id。LEFT JOIN保证主表数据不丢失
`dm_bill_event_charge_without_visit_v2_df`LEFT JOIN`track_event_info``e.event_name = event_info.name AND event_info.is_delete = &#x27;0&#x27;`获取事件中文名。使用field_mappings中的实际字段名name。is_delete=&#x27;0&#x27;过滤条件放在ON中，不影响主表行数，仅过滤维度表数据
`dm_bill_event_charge_without_visit_v2_df`LEFT JOIN`dwm_bill_optimize_event_v2_df``e.event_guid = opt.event_guid AND e.date = opt.date`通过event_guid和date双字段关联，标识可优化事件。LEFT JOIN保证主表所有事件行保留，关联成功的事件行用于计算可优化成本
`dm_bill_event_charge_without_visit_v2_df`LEFT JOIN`dwm_bill_event_onetrack_charge_share_di``e.app_id = dept.app_id AND e.date = dept.date`获取部门归属信息。此表作为维度表使用，已通过CTE按app_id,date去重。LEFT JOIN保证主表数据不丢失

#1 指标口径

**可优化成本的计算范围**：是否仅对与dwm_bill_optimize_event_v2_df关联成功的事件行求和？还是对主表所有事件行求和？

如果对主表所有事件行求和，则无法区分可优化与不可优化事件，看板失去"圈定可优化事件"的核心目标

建议仅对与dwm_bill_optimize_event_v2_df关联成功（即event_guid和date均匹配）的事件行求和，因为该表明确标识了所有可优化事件
#2 维度字段

**track_app_info表中未找到'app_name'字段**，上游key_fields为{'name', 'real_app_id', 'owner', 'project_id'}，当前使用'name'字段作为app_name

如果name字段不是应用名称，则app_name维度数据错误

建议确认track_app_info.name字段是否就是应用名称（app_name）
#3 维度字段

**track_app_info表中未找到'app_owner'字段**，上游key_fields为{'name', 'real_app_id', 'owner', 'project_id'}，当前使用'owner'字段作为app_owner

如果owner字段不是应用负责人，则app_owner维度数据错误

建议确认track_app_info.owner字段是否就是应用负责人（app_owner）
#4 维度字段

**track_event_info表中未找到'event_name_cn'字段**，上游key_fields为{'name', 'display_name', 'project_id'}，当前使用'display_name'字段作为event_name_cn

如果display_name字段不是事件中文名，则event_name_cn维度数据错误

建议确认track_event_info.display_name字段是否就是事件中文名（event_name_cn）
